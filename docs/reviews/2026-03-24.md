## Marvin's reaction

"I am doubting the overall wisdom of this whole daily briefing and what it's being asked to do. Could it be possible that we're asking it to do too much?"

This wasn't about today's briefing being bad — it was about the pattern across 15 sessions. Each sub-task (email curation, calendar, surprise, vault, day plan) is "a serious piece of work, requires effort, tokens." Asking one model to do all of them in one pass is why it never reliably gets all the things.

"With this OpenClaw loop, is it not the right way to be thinking to split all of these tasks out into their own individual tasks and they all just feed back into Jimbo?"

## What we did

### Phase 1: Distributed OpenClaw cron jobs

Upgraded OpenClaw from 2026.3.1 to 2026.3.23-2 (22 releases behind). Ran doctor, added startup optimizations. Created 6 new skills and 6 new OpenClaw cron jobs (email-scanner, calendar-briefing, vault-manager, surprise-game, morning-summary, accountability). Disabled old monolithic briefings. Removed model-swap crons.

Test run of email-scanner: worked, delivered to Telegram. But: **491K input tokens** on first run, **2M tokens** on second (persistent session accumulated history). Cost: $0.15-0.63 per run. At 8 runs/day, that's $1.20-5.00/day vs current $0.02-0.07.

### Phase 2: The token cost realisation

Root cause: OpenClaw injects 100-150K tokens of bootstrap context (SOUL.md, AGENTS.md, HEARTBEAT.md, etc.) on every agent turn. This is by design — the platform is built for rich conversational agents. But it means lightweight cron jobs have a massive fixed overhead.

**Key insight from Marvin:** "We're going round in circles with our decisions — we want to work with OpenClaw but we're also fighting it and circumventing it to keep costs down."

This led to the tiered architecture: most jobs don't need an LLM at all.

### Phase 3: Tiered automation plan

- **Tier 1:** Python scripts + Telegram Bot API. Zero LLM cost. For mechanical jobs (email alerts, calendar summary, vault status, morning status).
- **Tier 2:** OpenClaw + free Kimi K2. For conversational interaction and heartbeat nudges.
- **Tier 3:** Opus on dedicated always-on Mac. Free via Max plan. For creative work (surprise game, accountability, blog drafting).
- **Tier 4:** Dashboard on-demand. Manual triggers from site admin.

All 6 new cron jobs disabled pending Tier 1 Python replacement. Skills kept on VPS for when we re-enable creative jobs later.

## The pattern

This session crystallised a tension that's been building since session 2:

**OpenClaw is designed for rich conversational agents, not lightweight automation.** The bootstrap overhead (100K+ tokens/turn) is a feature for conversation — it gives the agent personality, memory, and context. But it's a cost problem for "check a score and post to Telegram."

We've oscillated between "use OpenClaw for everything" (sessions 1-8, expensive, unreliable) and "use Python for everything" (sessions 2-5, cheap, no personality). The tiered model accepts both are right — for different jobs.

The dedicated Mac with free Opus access is the missing piece. It provides unlimited creative intelligence without per-token cost, solving the "surprise game needs Opus but Opus costs money" problem from session 5.

## What's been built (sessions 16-17, between reviews)

Two significant systems have been built since session 15:

### Autonomous dispatch system (session 16)
- Dispatch queue in jimbo-api — propose, approve, execute, complete lifecycle
- M2 dispatch worker — Claude Code on always-on Mac pulls tasks, executes, opens PRs
- Dashboard at `/app/jimbo/dispatch` — dispatch queue UI with worker status, proposal cards, history with expandable details
- Agent templates (coder, researcher, drafter) with shared output contract
- Visual evidence upload to R2

### PR feedback loop (session 17)
- GitHub webhooks on jimbo-api — PR merge triggers dispatch completion, PR close with comment captures rejection reason
- Subtask rollup — dispatch items track parent/child relationships
- Template refactor — shared output contract extracted

### Dispatch flow split (today's design)
New design spec (`2026-03-26-dispatch-flow-split-design.md`) splits dispatch into three flows:

- **Commissions**: GitHub Issues labeled `ralph` → dispatched to M2 → PR back. Code work.
- **Quests**: Vault tasks only Marvin can do. Surfaced in briefings, nudged by Jimbo. Never dispatched.
- **Recon**: Research/intelligence gathering → dispatched to M2 → artifact committed directly + Telegram notification. No PR gate.

Key design decisions:
- GitHub is the source of truth for code tasks (not the vault)
- Vault becomes purely quests + recon + ideas
- `gh-issue` skill creates well-structured issues that already have the context agents need
- One combined approval surface (Telegram) for both commissions and recon

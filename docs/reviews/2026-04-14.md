## Root cause investigation

**`active_models: []` was a red herring.** The model IS configured (`openrouter/google/gemini-2.5-flash`), the OpenRouter key is healthy ($9.45 of $10 weekly remaining), memory is fine (1.1Gi available), Telegram API reachable via curl. The health endpoint just doesn't report the model from openclaw.json correctly.

**The real cause: Telegram long-polling broken since the 2026.4.9 update.**

Evidence chain:
1. **Apr 8:** Agent fully active — 14 activities, email scans every 2 hours, heartbeat, accountability report. Busiest day ever.
2. **Apr 9 08:51:** OpenClaw updated to 2026.4.9 (config-audit.jsonl confirms write at this timestamp).
3. **Apr 9 09:00:** Last agent activity — 2 email checks, 1 surprise game. Then silence.
4. **Apr 9 → Apr 14:** Zero agent activity. Pipeline cron keeps running, agent does nothing.
5. **Apr 13:** 181 service restarts in one day (log file shows 181 "starting..." entries). Mix of config-change restarts from control UI work and crash-restart loops. None resolved the issue.
6. **Apr 14 logs:** Only 3 entries — all `UND_ERR_SOCKET` Telegram errors. The gateway starts, the Telegram provider starts, but the polling connection never establishes.

The Telegram provider logs `[default] starting provider` but never logs a connection success. Cron scripts reach Telegram fine (direct Bot API calls via curl), but OpenClaw's Node.js HTTP client (undici) can't maintain the long-polling connection.

**OpenClaw 2026.4.12 is available** (running 2026.4.9). Three versions behind. The update may fix this.

## Pattern

4th time Jimbo has been fully dead (sessions 12, 16, 18, 19). Each time, the data pipeline keeps producing good output that nobody delivers. The gap between "data available" and "briefing delivered" remains the core failure mode.

New pattern: **platform updates break things silently.** The 2026.4.9 update on Apr 9 killed the Telegram connection with no alert, no error in the activity log, no notification to Marvin. Five days of silence before discovery. The system needs an "agent hasn't acted in N hours" alert — independent of the agent itself.

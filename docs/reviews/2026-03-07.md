## The core finding

**Without visibility of what's going on, it's difficult to judge these briefings meaningfully.**

Four sessions of reviews. Four sessions of "experiment tracker is empty, activity log is empty." The monitoring backbone has been broken since day one, and we've been trying to evaluate briefing quality blind.

The root cause: **two databases that don't talk to each other.**

- Sandbox scripts (`activity-log.py`, `cost-tracker.py`, `experiment-tracker.py`) write to SQLite files inside Docker
- The dashboard reads from jimbo-api, which has its own SQLite DB
- Nothing bridges them
- Data goes into the sandbox and dies there

The jimbo-api already has a working `/api/activity` endpoint with POST, GET, and rate. It just has no callers. The plumbing exists — nobody connected it.

## Calendar: not broken, ignored

Investigated the "12 events in pipeline, 0 in briefing" discrepancy. The calendar helper works. The events are in briefing-input.json (parkrun, football, Six Nations rugby, breakfast, Thomas Antony visit — 12 items). Jimbo said "the helper returned nothing" — a factual error. He had the data and didn't read it.

Previous sessions blamed calendar fabrication (salsa, "Breakfast" at 14:00). Today reveals the opposite problem: real data present, LLM claiming it's absent. The daily-briefing skill says "Calendar contains ONLY events from briefing-input.json" but Jimbo isn't parsing the JSON correctly in self-compose mode.

Note: the "Freshen up" and "Breakfast" items from previous sessions that we thought were hallucinations are REAL calendar events — they're in a subscribed calendar (`dd13ho8...`). The `--primary-only` flag doesn't filter them because that calendar has `accessRole: owner`.

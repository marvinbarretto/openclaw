## What we found: three root causes

### Root cause 1: Sandbox env var blocking (OpenClaw 2026.4.9 bug)

The 2026.4.9 update introduced `sanitizeEnvVars()` which blocks any env var matching `/_?(API_KEY|TOKEN|PASSWORD|PRIVATE_KEY|SECRET)$/i` from reaching Docker sandbox containers. This means `JIMBO_API_KEY`, `JIMBO_API_URL`, and all other operational keys defined in `openclaw.json` were silently stripped before reaching the agent's tools.

The agent could start, receive messages, and run — but every tool that needed API credentials failed silently. `health-helper.py`, `context-helper.py`, vault API calls, dispatch API calls — all broken.

GitHub Issue #25951 confirms this is a known bug. The `customAllowedPatterns` config option exists in the sanitizer but is never wired through to Docker container creation.

**Fix applied:** Two-file patch:
1. `sanitize-env-vars-CfT4R84s.js` — added `allowedSensitiveKeys` rescue path
2. `docker-CyNUQ0H6.js` — derives allowlist from config-defined env var keys

Also had to destroy and recreate the sandbox container — Docker injects env vars at creation time only, so the old container (created pre-patch) still had the stripped vars.

### Root cause 2: Wrong skill invocation

The agent picked `morning-summary` and `calendar-briefing` instead of `daily-briefing`. Even the internal cron job that explicitly says "Read and follow /workspace/skills/daily-briefing/SKILL.md" was ignored. The model never read the actual briefing skill — it freestyled from whatever pattern it had.

**Fix applied:**
- Updated `daily-briefing` skill description to include trigger words ("INVOKE THIS when user says good morning")
- Replaced SOUL.md's outdated "Morning Briefing Minimum Bar" section (which still said "propose day plan", contradicting session 17's reporter rewrite) with explicit "Skill Routing (CRITICAL)" section

### Root cause 3: Poisoned session context

The killer finding. Session file `3701e0c4-6433-4f8d-a80b-9c3a9754b476.jsonl` from April 13 contained fabricated task data — "Plan Weekend Hike", "Organize Digital Photos", "Research New Coffee Maker" — invented by a model that couldn't reach real data (root cause 1). Every subsequent model inherited this via `heartbeat.target: "last"`, which routes all conversation to the last active session.

This explains the "strange" pattern of 4 different models (Gemini Flash, Gemma4, GPT-OSS, and others) all producing identical fabricated responses. They weren't independently hallucinating — they were all reading the same poisoned session history and echoing it back.

**Fix applied:** Archived poisoned session to `.poisoned.bak`.

## Key insights

**The three root causes compounded.** Env var blocking (1) meant the agent couldn't reach real data. Without data, the model fabricated responses (2, 3). The fabricated responses got cached in the session file. Every new model read the cached fabrication and repeated it. The system settled into a stable failure state — consistently wrong, never correcting.

**Session poisoning is a new failure mode.** We've seen stale context in memory files (session 7), stale SOUL.md (session 18), and stale pipeline data. But a poisoned *session* — where fabricated data becomes the "ground truth" for all subsequent models — is worse because it's invisible. There's no file to audit; it's buried in a binary-ish JSONL stream.

**Patching vendor code is fragile.** Both patches will be overwritten on next OpenClaw update. Need either: (a) upstream fix, (b) a post-update script that re-applies patches, or (c) an env var validation check in the pipeline.

**"It's not about the model" — confirmed.** This session definitively proved that identical bad output from 4 different models is a systems problem, not a model quality problem. The session 7 pattern ("check the plumbing before blaming the model") continues to be the most important lesson.

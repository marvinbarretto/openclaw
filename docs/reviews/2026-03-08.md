## What we learned

**The architecture was the problem, not the prompts.** Seven pipeline stages across three models and two API providers produced nothing useful. One capable model reading everything in one pass produced the best briefing we've seen.

The multi-stage pipeline (Flash triage → Haiku reader → Sonnet/Kimi compose) was designed to save money by using cheaper models for bulk work. But:
- Flash rejected everything (0 shortlisted, 5 sessions running)
- Haiku had nothing to read (0 gems, because 0 shortlisted)
- Sonnet/Kimi composed from empty data
- The hourly heartbeat burned tokens for "no nudge needed" messages
- The whole thing cost more ($10+/week) than doing nothing

Meanwhile, Opus via Max plan is free, sees all the data, and has the judgment to pick what matters.

**Over-engineering was the real enemy.** The system had: email triage worker, newsletter reader worker, experiment tracker, activity logger, cost tracker, model routing (Kimi ↔ Sonnet swaps), cron pipeline orchestrator, Telegram alerts, heartbeat checks, accountability reports. Most of this infrastructure was producing noise, not value.

## The path forward

Marvin wants to simplify radically. The plan discussed:

1. **Disable OpenClaw heartbeat and briefing delivery on VPS** — stop the token burn
2. **Keep VPS as silent data collector** — gmail-helper.py and calendar-helper.py on cron (cheap, already works), jimbo-api for dashboard
3. **Compose briefings locally with Opus** — `claude -p` with a simple prompt, free via Max plan
4. **Iterate on prompt quality locally** — fast feedback loop, zero cost

Also explored: Obsidian + OpenClaw "second brain" community patterns. Key finding from research: the people getting value from this use one good model reading a structured vault of markdown files. No multi-stage pipelines. Marvin already has the vault (13,000 notes), the context files, and the email/calendar data. He's been building plumbing when he needed a better reader.

Marvin noted he's open to using Obsidian as a vault UI but also questioned whether his custom site UI could serve the same purpose — and it can. Obsidian is just a UI layer over markdown files.

## Patterns across 5 sessions

- **Sessions 1-4:** Fixing plumbing. Workers not running → deploy workers. Calendar broken → fix API. Logging broken → migrate to API. Each fix revealed the next problem.
- **Session 5:** Stepped back. The plumbing was the problem.
- **Email quality:** The one thing that improved (session 3 email picks were "pretty good") was when Sonnet did its own triage from raw data — bypassing the worker pipeline entirely. Today's Opus test confirms: a good model with all the data beats a pipeline of cheaper models.
- **Calendar:** Four different failure modes across sessions (403, fabricated, ignored, time math errors). Calendar data is reliable in the pipeline JSON — the problem is always the model's handling of it. Opus got it right on the first try.
- **Cost:** The system designed to save money ($15-25/month budget) hit its credit limit doing nothing useful. The free alternative (Opus via Max) produced the best result.

## What we learned

### OpenClaw showcase research

Reviewed openclaw.ai/showcase and awesome-openclaw-usecases. The pattern across successful agents: **step-by-step tool use, conversational, agentic autonomy.** Not one-shot composition from giant JSON files.

Step 3.5 Flash is the most popular model on OpenClaw via OpenRouter. Other people get good results from it. Our problem isn't the model — it's how we're using it.

### The real diagnosis

We've been asking Flash to do one complex step (read 25KB JSON, determine mode, apply personality, follow 6 hard rules, compose editorial). OpenClaw is built for many simple steps (fetch data, send a message, fetch more data, respond to questions).

We moved all the intelligence outside OpenClaw (cron scripts, Opus on Mac) and reduced Jimbo to a template relay. That's the opposite of what OpenClaw is for.

### The course change

**Opus produces structured data (no voice).** Publishes to jimbo-api endpoint, not a file.

**Jimbo is a full agent again.** Fetches analysis from API. Delivers section by section (one tool call per step). Stays in conversation for follow-ups. Can act: create calendar events, browse vault, fetch email detail.

**Jimbo owns the voice.** SOUL.md personality applied by Jimbo, not baked into Opus output. Consistent voice whether delivering briefing or answering follow-ups.

Design spec written: `docs/superpowers/specs/2026-03-16-briefing-api-delivery-design.md`

### The bind mount discovery

Live testing failed — Jimbo never triggered the new skill. Both Flash and Haiku ignored it. Investigation revealed:

```
"/usr/lib/node_modules/openclaw/skills:/home/openclaw/.openclaw/workspace/skills:ro"
```

A Docker bind mount in `openclaw.json` was mapping OpenClaw's **bundled skills** over our custom skills directory. Our skills were invisible to the model for all 7 sessions. Every time we pushed skills to the VPS, they were hidden behind this mount.

**Fix:** Removed the bind mount. After restart + sandbox recreation, skills loaded correctly (confirmed in logs).

### Model self-tagging

Haiku tagged itself as `[Flash]` — SOUL.md had a rule asking models to self-report their model name, but they can't reliably detect it. Removed the rule.

### Final test result (Flash, with bind mount fix)

The calendar fabrication problem that plagued every session appears to be solved by the bind mount fix alone.

### Root causes identified this session

1. **Bind mount hiding custom skills** — 7 sessions of invisible skills. The single biggest fix.
2. **Model self-tagging unreliable** — removed.
3. **Skill not triggering API fetch** — the model reads raw pipeline data instead of fetching from the API. Lower priority now that calendar fabrication is solved.

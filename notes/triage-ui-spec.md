# Notes Triage UI — Architecture Spec

## The Problem

~6,500 unprocessed notes in a vault inbox. We have an LLM classifier (Gemini Flash) that gets ~70% right. Need a mobile-friendly UI to review the LLM's suggestions and confirm/override them — on the bus, on the phone, wherever.

## Core Insight: Separate Decisions from File Moves

The UI doesn't touch the vault filesystem. It collects triage decisions against a pre-built manifest. Decisions are applied later on the laptop.

## Architecture

```
LAPTOP (openclaw repo)                         VPS (API + UI)
─────────────────────                           ──────────────
1. process-inbox.py --manifest
   → generates manifest.json
   (filename, title, preview, suggested
    type/tags/confidence for each item)

2. rsync manifest.json → VPS                   3. Hono API serves manifest
                                                   GET  /api/triage/queue
                                                   POST /api/triage/decisions

                                                4. React UI on /app/notes-triage
                                                   Shows cards, collects swipes
                                                   Writes decisions to API

5. Pull decisions.json from VPS
6. apply-decisions.py
   → calls review_helper.py for each
   → moves files, updates frontmatter
```

No vault on VPS. No real-time file moves. Just two JSON files going back and forth.

## UI Design — Card-Based Swipe Interface

### Default view: one card at a time

```
┌─────────────────────────────────┐
│  12/6511                    ⚙️  │  ← progress counter
│                                 │
│  Dopesic                        │  ← title (large)
│  google-keep · 558d ago         │  ← source + age
│                                 │
│  "Dopesic"                      │  ← preview (body text)
│                                 │
│  ┌─────────────────────────┐    │
│  │ 🤖 Gemini suggests:     │    │  ← LLM suggestion box
│  │ media [tv, drama,       │    │
│  │   project:film-planner] │    │
│  │ confidence: 9           │    │
│  └─────────────────────────┘    │
│                                 │
│  ┌──────┐ ┌──────┐ ┌────────┐  │
│  │  ✓   │ │  ✗   │ │  Edit  │  │  ← action buttons
│  │Accept│ │Archiv│ │Override│  │
│  └──────┘ └──────┘ └────────┘  │
│                                 │
│         [ Skip → ]              │  ← skip button (subtle)
└─────────────────────────────────┘
```

### Interaction model

**For high-confidence items (confidence >= 8):**
- **Swipe right / tap Accept** → confirm LLM suggestion as-is. Next card.
- **Swipe left / tap Archive** → archive with default stale_reason. Next card.
- **Tap Edit** → expand the override form (see below)
- **Swipe up / tap Skip** → skip, come back later

**For low-confidence / needs-context items:**
- Show the card but with a yellow/amber indicator
- No pre-filled suggestion or a weaker one
- User must tap Edit to classify, or Skip

**Keyboard shortcuts (desktop):**
- `D` — accept (direct, confirm suggestion)
- `A` — archive (with most common stale_reason)
- `E` — edit/override
- `S` — skip
- `←` `→` — swipe left/right
- `U` — undo last decision

### Override form (expands on Edit tap)

```
┌─────────────────────────────────┐
│  Action:  ○ Direct  ○ Archive   │
│           ○ Context ○ Skip      │
│                                 │
```

### Tag autocomplete

Pre-populated from:
- Tags already used in classified notes (build frequency list from manifest)
- Project tags: `project:localshout`, `project:film-planner`, `project:spoons`, `project:openclaw`
- Person tags: `person:*` (free text after prefix)
- Common tags from the type taxonomy (e.g. selecting type=travel suggests country/city tags)

## Type Taxonomy

Selectable in dropdown:

| Type | Description |
|------|-------------|
| bookmark | Saved URL/link |
| recipe | Food/drink recipe or recommendation |
| media | Film, TV, music, podcast, book |
| travel | Destination, deal, trip idea |
| idea | Personal thought, project idea |
| reference | Factual info (address, how-to, code) |
| event | Specific event with date/time |
| task | Action item or to-do |
| checklist | List of items |
| person | Contact info, someone to remember |
| finance | Money, budgeting, deals |
| health | Fitness, nutrition, medical |
| quote | Saved quote or passage |
| journal | Personal reflection, diary entry |
| political | Political commentary or opinion |

## Stale Reasons

Selectable when action = archive:

| Reason | When to use |
|--------|-------------|
| stale | Too old to be relevant |
| dead-url | URL is broken/dead |
| completed | Task/action already done |
| duplicate | Duplicate of another note |
| past-event | Event has already happened |
| empty | No meaningful content |

## API Endpoints

### `GET /api/triage/queue?offset=0&limit=50`

Returns a page of manifest items, excluding already-decided items.

Response: `{ items: [...], total: 6511, remaining: 6489, decided: 22 }`

### `POST /api/triage/decisions`

Body: `{ decisions: [{ id, action, type?, tags?, stale_reason?, title?, context? }] }`

Appends to decisions store. Returns: `{ saved: 3, total_decided: 25 }`

### `GET /api/triage/stats`

Returns: `{ total: 6511, decided: 25, remaining: 6486, by_action: { direct: 15, archive: 8, skip: 2 } }`

### `POST /api/triage/undo`

Body: `{ id: "note_85ded1a" }`

Removes the most recent decision for that note. Returns it to the queue.

## Scripts to Build (openclaw repo)

### `process-inbox.py --manifest`

New flag. Instead of classifying and moving files, outputs `data/triage-manifest.json`. Runs Gemini Flash on each item, collects suggestions. Can use `--skip-fetch` and `--limit` flags.

### `scripts/push-manifest.sh`

Rsyncs `data/triage-manifest.json` to VPS.

### `scripts/pull-decisions.sh`

Rsyncs decisions JSON from VPS to `data/triage-decisions.json`.

### `scripts/apply-decisions.py`

Reads `data/triage-decisions.json`, applies each decision using `review_helper.py`:
- direct → `review_helper.py move <file> data/vault/notes '<updates>'`
- archive → `review_helper.py move <file> data/vault/archive '<updates>'`
- context → `review_helper.py context <file> '<text>'`
- skip → no-op

Prints summary: "Applied 150 decisions: 95 direct, 40 archive, 10 context, 5 skip"

## Deployment

- **API**: Hono on VPS, behind Caddy reverse proxy at `api.marvinbarretto.com/triage/*` or similar
- **UI**: React island in Astro site at `/app/notes-triage`, behind Cloudflare Access
- **Auth**: API key in `X-API-Key` header (sufficient behind Cloudflare Access)
- **Manifest**: pushed from laptop via rsync when you run a new classification batch
- **Decisions**: pulled to laptop when you're ready to apply

## Future Enhancements

- **Batch mode**: "archive all items with confidence >= 9 and suggested action = archive" — one tap
- **Filters**: by source, age, confidence level, has-url, type
- **Undo history**: see and revert recent decisions
- **Offline mode**: cache manifest in service worker, sync decisions when back online
- **URL preview**: for bare URL notes, show fetched content inline (oEmbed for tweets, page title + screenshot for others)
- **Stats dashboard**: progress over time, accuracy of LLM suggestions vs your overrides
- **Feedback loop**: export overrides as training data to improve process-inbox.py prompt
